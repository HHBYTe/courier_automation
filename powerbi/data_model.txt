Courier Automation — Power BI data model
==========================================

Star schema. Two fact tables (shipments, refunds), three dimensions.

Tables to load in Power BI Desktop
----------------------------------

Fact:
  unified_shipments   — unified/output/unified_shipments.parquet  (~370k rows)
  refunds             — unified/output/refunds.parquet            (~3.3k rows)

Dimension:
  dim_date            — powerbi/output/dim_date.parquet
  dim_carrier         — powerbi/output/dim_carrier.parquet
  dim_service_class   — powerbi/output/dim_service_class.parquet

Measures:
  Measures            — a dummy in-Power-BI table created via Home →
                        Enter Data (one column, name = "_"). Mark the
                        column as Hidden after creating the table.
                        All measures from measures.dax live here.


Relationships
-------------

  unified_shipments[posting_date]       → dim_date[date]            (many:1, active)
  unified_shipments[carrier]            → dim_carrier[carrier_code] (many:1)
  unified_shipments[service_class]      → dim_service_class[service_class] (many:1)

  refunds[posting_date]                 → dim_date[date]            (many:1, active)
  refunds[carrier]                      → dim_carrier[carrier_code] (many:1)

  No direct relationship between refunds and unified_shipments.
  Drill-down from a refund to its shipment uses (carrier, shipment_id)
  as a measure-time filter pair.

  All relationships: cross-filter direction = single (from dim → fact).


Column-level settings
---------------------

unified_shipments:
  - año, mes:                 hide (use dim_date)
  - invoice_date, posting_date: format "yyyy-mm-dd", mark posting_date
                              as the date in Mark as Date Table for
                              relationships if you skip dim_date
  - bultos_count, weight_kg, base_cost, fuel_surcharge, other_surcharges,
    total_net, total_net_eur, base_cost_eur, fuel_surcharge_eur,
    other_surcharges_eur:     summarisation = Sum
  - fx_rate_to_eur:           summarisation = Do not summarise
  - shipment_id, invoice_id:  summarisation = Do not summarise
  - source_file:              hide (debug only)
  - native cost columns (total_net, base_cost, …) can be hidden if you
    only ever report in EUR — the *_eur columns are what the measures use

refunds:
  - same as shipments

dim_date:
  - Table Tools → Mark as Date Table → date column
  - sort `month_name` by `month`, `month_name_short` by `month`,
    `day_name` by `day_of_week`


Currency
--------

All cost measures are in EUR. The pipeline converts every non-EUR
carrier (UPS = GBP, WWEX = USD) to EUR with a single frozen rate
(see unified/fx_rates.py) and stores the result in the Shipments[*_eur]
columns. Cost measures read those columns, so they sum cleanly across
all 7 carriers with no slicer needed.

A `currency` slicer is now OPTIONAL — only useful as an informational
filter (e.g. "show me just the GBP-native carrier"). It is no longer a
safety guard. The native-currency columns (total_net, base_cost, …) are
still in the model for audit if anyone needs the original figures.


Suggested 5-page operational drill-down
---------------------------------------

Page 1 — Overview
  Cards:    Total Shipments • Total Bultos • Gross Cost • Net Cost
  Trend:    Gross Cost by year_month, line + column combo
  Mix:      Total Shipments by carrier (donut)
  Slicers:  dim_date[year]  (currency slicer optional — see Currency note)
  Filter:   none

Page 2 — Carrier Detail
  Slicer:   single-select carrier dropdown (top)
  Cards:    Total Shipments • Cost per Shipment • Fuel % of Net •
            Refund Rate %
  Visual:   Gross Cost by year_month (line), with Gross Cost LY overlay
  Table:    service_raw × Total Shipments × Gross Cost × Cost per Bulto
  Drill:    enable drill-through to Page 5 (Refunds) on right-click

Page 3 — Service Mix
  Matrix:   carrier (rows) × service_class (columns), values = Total Shipments
  100%-stacked bar:  Gross Cost by service_class, broken down by carrier
  Card:     Active Service Classes count
  Slicers:  year, month

Page 4 — Destinations
  Map:      destination_country (filled map), size = Total Shipments
  Bar:      Top 15 destinations by Gross Cost (filter Top N)
  Table:    destination_country × Total Shipments × Total Bultos ×
            Avg Weight per Shipment × Gross Cost × Cost per Bulto
  Slicers:  carrier
  Note:     SEUR destination_country is derived from its plaza codes
            (see unified/normalizers/seur.py + data/seur/other/Destino.csv)
            and is ~100% populated — SEUR DOES land on the map now.
            Royal Mail invoices don't carry destination_country either;
            we default RM rows to "GB" in the normalizer because every
            observed RM service in this dataset is UK-domestic (1st/2nd
            class, Tracked 24/48). If a future RM invoice contains an
            international service, revisit this default.

Page 5 — Refunds
  Cards:    Refund Total • Refund Rate %
  Trend:    Refund Total by year_month (column)
  Table:    refunds[shipment_id] × refunds[posting_date] × refunds[total_net]
            with carrier filter on the page
  Drill:    "View original shipment" — right-click to drill back to Page 2
            filtered by (carrier, shipment_id)


===================================================================
HOW TO RUN EVERYTHING
===================================================================

There are two scripts and they MUST run in this order. Both are run
from the repo root in PowerShell. The .pbix never needs re-authoring
once built — you only ever re-run the scripts and hit Refresh.

  Script 1:  python -m unified.build
             Reads  data/<carrier>/*.parquet  (all 7 carriers)
             Writes unified/output/unified_shipments.parquet
                    unified/output/refunds.parquet
                    unified/output/rejections.parquet
                    unified/output/manifest.json

  Script 2:  python -m powerbi.build_dims
             Reads  unified/output/unified_shipments.parquet
             Writes powerbi/output/dim_date.parquet
                    powerbi/output/dim_carrier.parquet
                    powerbi/output/dim_service_class.parquet

Script 2 reads Script 1's output, so unified.build ALWAYS runs first.


-------------------------------------------------------------------
A. Routine data refresh  (the common case — new invoices arrived)
-------------------------------------------------------------------

  1. python -m unified.build
  2. python -m powerbi.build_dims
  3. Open the .pbix → Home → Refresh (the refresh icon)
  4. Save the .pbix

That's it. Verify step 1 printed all 7 carriers and a kept-row count
near ~370k (see "Sanity check" below). If a carrier is missing or the
count is way off, DO NOT refresh Power BI — fix the build first.


-------------------------------------------------------------------
B. First-time setup  (building the .pbix from scratch)
-------------------------------------------------------------------

  1. python -m unified.build
  2. python -m powerbi.build_dims
  3. Power BI Desktop → Get Data → Parquet → load all 5 output files
  4. Modeling → wire the 5 relationships listed above
  5. Mark dim_date as a Date Table (Table Tools → Mark as Date Table)
  6. Home → Enter Data → table "Medidas" (one column "_", mark hidden)
  7. Paste each block from measures.dax as its own measure into Medidas
  8. Add the calculated columns (Direction, Weight Bucket — see below)
  9. Build the 5 report pages
  10. Save the .pbix


-------------------------------------------------------------------
C. After changing the FX rates  (unified/fx_rates.py)
-------------------------------------------------------------------

The frozen GBP/USD→EUR rates live in unified/fx_rates.py. They feed
the *_eur columns. After editing them you must rebuild so the new
rates flow into the parquet:

  1. Edit unified/fx_rates.py  (update RATE_TO_EUR and FROZEN_RATE_AS_OF)
  2. python -m unified.build
  3. python -m powerbi.build_dims      (not strictly needed, but harmless)
  4. Open the .pbix → Home → Refresh → Save

dim_carrier / dim_service_class only change if you add a carrier or a
service class — otherwise step 3 is a no-op you can skip.


-------------------------------------------------------------------
!!! GOTCHA: never leave a partial build on disk !!!
-------------------------------------------------------------------

unified.build accepts an optional  --carriers  flag for testing a
single carrier, e.g.:

  python -m unified.build --carriers ups        # UPS ONLY — partial!

This OVERWRITES unified/output/ with just that carrier's rows. If you
then refresh Power BI, the report silently loses the other 6 carriers
(symptom: visuals go blank or near-empty for everything except the one
carrier you tested).

RULE: any time you run --carriers, the very next thing you do is run a
full  python -m unified.build  (no flag) before touching Power BI.
When in doubt, just run the full build — it only takes ~16 seconds.


-------------------------------------------------------------------
Sanity check  (what a healthy `python -m unified.build` looks like)
-------------------------------------------------------------------

The run log should list ALL 7 carriers and end near these numbers:

  correos    ~58,800 kept       seitrans   ~2,100 kept
  seur      ~267,300 kept       dachser      ~420 kept
  spring     ~1,200 kept        ups       ~18,000 kept
  wwex      ~22,600 kept
  -> ~370,000 kept rows, ~3,300 refunds, ~16,500 rejected

Currency by carrier: UPS = GBP, WWEX = USD, all others = EUR.
total_net_eur should have zero nulls. If the build hits a currency
with no rate in fx_rates.py it aborts with a clear error — add the
rate and re-run.


-------------------------------------------------------------------
Calculated columns to add in Power BI  (not in the parquet)
-------------------------------------------------------------------

  Shipments[Direction]          Domestic / International / Unknown
  Shipments[Weight Bucket]      "001 KG".."MÁS 200 KG" / "Sin peso"
  Shipments[Weight Bucket Sort] integer sort key for Weight Bucket

These live in the .pbix and survive a data Refresh — you only author
them once (first-time setup step 8).
